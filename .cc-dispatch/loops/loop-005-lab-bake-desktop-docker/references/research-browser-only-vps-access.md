# Objective
A practical survey of methods for accessing a remote Linux VPS — SSH shell, a web-based admin UI served on a non-standard port, an editor/IDE workspace, and HTTP preview URLs — from a heavily locked-down client machine that cannot install client software or kernel-level VPN tooling. The constraint: corporate-managed laptops frequently block installation of VPN clients like Tailscale (which needs a TUN interface or special permissions), restrict arbitrary outbound ports, and run strict HTTPS-inspecting proxies. The architectural strategy under evaluation is to place all tunneling and serving software on the VPS side (fully controlled by the operator) so the client only needs a modern web browser. This research evaluates the leading approaches — VS Code Remote Tunnels (code tunnel) via the vendor relay, Cloudflare Tunnel with Cloudflare Access, browser-served IDEs (code-server, openvscode-server), browser-based SSH terminals (ttyd, wetty, sshwifty), and Tailscale Funnel/Serve as a server-side-only option — on setup complexity, what survives strict corporate proxies and firewalls, authentication and security model, and reliability for a time-boxed workshop where many users connect at once.
### Context
This research supports a hands-on workshop where each participant operates a remote Linux VPS running an AI agent. The default access plan uses Tailscale for secure networking, but some participants will be on heavily locked-down corporate machines that cannot install a VPN client. We need browser-only fallback paths for every access need (SSH shell, the agent's web Control UI on a non-standard port, a VS Code editing experience, and HTTP preview URLs) that survive strict corporate proxies and firewalls. All serving/tunneling software can be installed on the VPS, which the operator fully controls; the participant's machine should need nothing beyond a modern browser.
### Focus Areas
**VS Code Remote Tunnels (code tunnel) via Relay** — How does the VS Code 'code tunnel' / Remote Tunnels feature work, where the tunnel-host binary runs on the remote VPS and the user connects through the vendor relay using [vscode.dev](http://vscode.dev) in a browser with no local install? What is the authentication flow (GitHub/Microsoft login), what outbound connectivity does the VPS need, and crucially what does the CLIENT need — does it work entirely in a browser behind a strict corporate proxy that only allows HTTPS to well-known domains? What are the known limitations, relay reliability concerns, and any enterprise-network failure modes? Is the relay traffic distinguishable/blockable by corporate firewalls?
**Cloudflare Tunnel + Cloudflare Access** — How does running cloudflared on the VPS to expose internal services (a web UI on a non-standard port, HTTP preview apps, and optionally SSH) as public HTTPS hostnames work, gated by Cloudflare Access with email one-time-PIN authentication? What is the setup on the server side, what does the end user experience (browser only, email OTP), and how well does this survive strict corporate proxies since traffic is standard HTTPS to Cloudflare edge? Can Cloudflare Access broker SSH to a browser terminal? What are the free-tier limits and the security considerations of exposing services this way for a short-lived workshop?
**Browser-Served IDEs and SSH Terminals** — Compare browser-served development environments that run on the VPS and present a full IDE or shell in the browser over HTTPS: code-server and openvscode-server (full VS Code in the browser), and browser-based SSH terminals like ttyd, wetty, and sshwifty. For each: setup complexity on the VPS, authentication options, resource overhead, feature parity with desktop VS Code (extensions, Remote-SSH-like file access), and suitability as a fallback when Remote Tunnels is blocked. Which combination gives a locked-down-client user the closest experience to a native editor + terminal using only a browser?
**Server-Side Tailscale (Funnel/Serve) and Decision Matrix** — Can Tailscale be used in a way that requires NO client install on the user's machine — specifically by running Tailscale on the VPS and using Funnel or Serve to expose a web UI publicly over HTTPS so a browser-only user can reach it? What are the tradeoffs (public exposure, auth, Funnel limitations) versus a full tailnet with clients installed? Finally, synthesize a decision matrix: for each access need (interactive shell, web admin UI on a custom port, code editor, HTTP preview URL), what is the recommended browser-only method, the recommended method when the client CAN install software, and the fallback order. The goal is that a user on a Chromebook or fully locked-down corporate laptop can complete an entire hands-on workshop using only a browser.
### Desired Output
A practical report with a per-method evaluation, an honest assessment of what survives strict corporate proxies, security/auth notes for each, and a clear decision matrix mapping each access need to a recommended browser-only method plus fallback order. Specific enough to implement on the VPS provisioning side.
### Out of Scope
Self-hosted VPN servers requiring client installation
Solutions requiring admin rights on the client machine
Mobile-only access
Long-term production hardening beyond workshop timescale
# Report
### Executive Summary
**Practical goal** — Provide a set of reliable, browser-only access patterns so workshop participants on locked-down corporate laptops (no client installs, strict HTTPS-inspecting proxies, limited outbound ports) can reach a remote Linux VPS for shell work, a hosted code editor, admin UIs on custom ports, and HTTP preview URLs. The architectures evaluated intentionally push all tunneling and serving onto the VPS side so the client only needs a modern browser.
**Short recommendation** — For the broadest success when only a browser is available: prefer Cloudflare Tunnel + Cloudflare Access for public HTTPS-protected web UIs and HTTP previews (best survivability through corporate proxies) and use browser-served IDEs (code-server or openvscode-server) or browser SSH terminals (ttyd/wetty/sshwifty) behind a Cloudflare Tunnel when you need an in-VPS editor or raw terminal. VS Code Remote Tunnels (vendor relay) gives the closest integrated VS Code experience in-browser but is fragile in strict enterprise environments because its relay domains are easily blocked and the service depends on WebSocket/HTTPS to vendor endpoints. Server-side Tailscale Funnel/Serve provides simple HTTPS exposure of services without client installs but is dependent on tailnet policies and Funnel port limitations; it generally traverses corporate firewalls but may be blocked by policies that filter unknown outbound endpoints. Recommendations below map each access need to the preferred browser-only and client-installed options and ordered fallbacks with concrete server-side provisioning notes.
### VS Code Remote Tunnels (code tunnel) via the vendor relay
**How it works** — The operator runs the tunnel/host component on the VPS, which registers itself with Microsoft's relay infrastructure; users connect from a browser at [https://vscode.dev](https://vscode.dev) to the relay and authenticate with GitHub or Microsoft to authorize the browser-to-relay-to-host connection. The tunnel is outbound-initiated from the VPS so no inbound ports need opening on the host side [Visual Studio Code Remote Tunnels](https://code.visualstudio.com/docs/remote/tunnels).
**VPS outbound requirements** — The host needs outbound HTTPS/WebSocket connectivity to Microsoft relay endpoints (examples include domains under [rel.tunnels.api.visualstudio.com](http://rel.tunnels.api.visualstudio.com) and [devtunnels.ms](http://devtunnels.ms)), and generally communicates over TCP/443 with WebSocket (wss) or HTTP/2 to vendor relays [Visual Studio Code Remote Tunnels](https://code.visualstudio.com/docs/remote/tunnels).
**Client requirements (browser-only viability)** — The client requires only a modern browser and the ability to reach the vendor relay and authentication endpoints (GitHub or Microsoft login pages). In an environment that allows HTTPS to major identity providers and the relay domains, users can connect from [vscode.dev](http://vscode.dev) without local software. However, any corporate proxy or firewall that blocks the tunnel relay domains or that interferes with WebSocket upgrades will break the in-browser flow [Visual Studio Code Remote Tunnels](https://code.visualstudio.com/docs/remote/tunnels) and real-world failure reports show relay connection timeouts and inability to connect from [vscode.dev](http://vscode.dev) despite the host reporting online [GitHub issue: connection failures](https://github.com/microsoft/vscode/issues/185311).
**Authentication and security model** — Authentication is via GitHub or Microsoft accounts (OAuth flows in the browser). The vendor relay trusts those identity assertions and authorizes the browser session to the host. Enterprises can control tunnel use via GPO/tenant policies and Microsoft publishes controls to restrict Dev Tunnel usage to tenants or disable anonymous access [Visual Studio Code Remote Tunnels](https://code.visualstudio.com/docs/remote/tunnels).
**What survives strict corporate proxies/firewalls** — If the corporate environment allows HTTPS (TCP/443) to the Microsoft relay domains and permits WebSocket upgrades, VS Code Remote Tunnels can work. If traffic inspection or domain blocking targets the relay domains (for example blocking [global.rel.tunnels.api.visualstudio.com](http://global.rel.tunnels.api.visualstudio.com)) or strips WebSocket upgrade headers, the service will fail; these domains are distinct and can be detected/blocked by enterprise firewalls [Visual Studio Code Remote Tunnels](https://code.visualstudio.com/docs/remote/tunnels) and observed failure reports [GitHub issue: #185311](https://github.com/microsoft/vscode/issues/185311).
**Reliability for workshops** — Provides the closest browser VS Code experience (actual VS Code server on the host) with integrated terminals, extensions and port forwarding, but in enterprise workshops hundreds of concurrent browser sessions may stress relay reliability and any transient relay issues or domain-level blocks affect all users. Community reports note timeouts and stale host sessions requiring restarts [GitHub issue: relay reliability reports](https://github.com/microsoft/vscode/issues/185311).
### Cloudflare Tunnel (cloudflared) + Cloudflare Access
**How it works** — Run cloudflared on the VPS as an outbound connector to Cloudflare’s edge. Configure a Cloudflare hostname (or use a Cloudflare-generated hostname) that maps to a local service; Cloudflare Access can gate that hostname with SSO or one-time PIN (email OTP) flows so users authenticate with a browser before reaching the origin [Cloudflare Tunnel setup](https://developers.cloudflare.com/tunnel/setup) and [Cloudflare One Access OTP](https://developers.cloudflare.com/cloudflare-one/integrations/identity-providers/one-time-pin).
**VPS outbound requirements** — cloudflared maintains outbound connections to Cloudflare’s network; it favors HTTP/2 or QUIC over specific endpoints (commonly speaking cloudflared uses port 7844 for tunnel protocols and falls back to HTTPS as needed). For restrictive firewalls you must allow connections to Cloudflare’s tunnel relay endpoints and optionally to [api.cloudflare.com](http://api.cloudflare.com) for management operations [Tunnel with firewall guidance](https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/configure-tunnels/tunnel-with-firewall).
**Client requirements (browser-only viability)** — The client only needs a browser that can reach Cloudflare’s public HTTPS edge. The Access gateway presents a browser-based auth flow (SSO or email OTP); Cloudflare’s one-time PIN option allows temporary access to users without pre-provisioned identity accounts, which is convenient for time-boxed workshops [Cloudflare One OTP docs](https://developers.cloudflare.com/cloudflare-one/integrations/identity-providers/one-time-pin).
**What survives strict corporate proxies/firewalls** — Cloudflare Tunnel is one of the strongest survivors in locked-down environments because clients simply speak HTTPS to Cloudflare’s global edge; corporate proxies that allow HTTPS to popular CDNs and cloud providers generally allow Cloudflare traffic. The main caveat is that cloudflared may prefer QUIC/HTTP/3 (UDP) or HTTP/2 over a nonstandard port and some enterprises block QUIC/UDP or nonstandard outbound ports — cloudflared can be configured to force HTTP/2 over TCP/443 as a workaround, but behavior depends on the proxy and environment [cloudflared behind HTTPS proxy community discussion](https://community.cloudflare.com/t/using-cloudflared-behind-https-proxy/637037) and official firewall guidance [Tunnel with firewall guidance](https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/configure-tunnels/tunnel-with-firewall).
**Authentication and security model** — Cloudflare Access sits in front as a Zero Trust gateway. You can require SSO (Okta, Azure AD, Google) or allow email OTP for ephemeral workshop access. Access can also enforce device posture and fine-grained policies per hostname or path [Cloudflare Access docs](https://developers.cloudflare.com/cloudflare-one/). Session-based protections, logging, and WAF can be applied at the edge before traffic reaches the VPS.
**SSH and browser terminals** — Cloudflare supports browser-based SSH sessions proxied through the Tunnel + Access stack, enabling browser SSH to origin servers protected by Access without a client-side SSH client, which is useful when clients are locked down [Cloudflare browser-based SSH use case](https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/use-cases/ssh).
**Free-tier and workshop constraints** — Cloudflare offers free-tier Tunnel/Access capabilities with reasonable limits for small-scale workshops. Quick tunnels are possible for local dev but have concurrency and feature limits; consult account-level limits if you expect large numbers of simultaneous connections [Cloudflare account limits](https://developers.cloudflare.com/cloudflare-one/account-limits).
**Reliability for workshops** — Cloudflare’s large global edge and mature infrastructure mean tunnels are resilient and usually transit well through corporate proxies. The Access OTP flow is browser-friendly for ephemeral attendees, though OTP email delivery delays can create minor friction (plan for short buffers and pre-notification) [OTP delivery community notes](https://community.cloudflare.com/t/not-receiving-otp-emails-in-cloudflare-zero-trust-one-time-pin-login/835879).
### Browser-served IDEs: code-server and openvscode-server
**How they work (server-side)** — Both projects run a VS Code-compatible server on the VPS and serve the VS Code UI in the browser over HTTPS/WebSocket. code-server is a maintained productized server with built-in auth options and production-focused docs, while openvscode-server is a lighter upstream-aligned server suitable for DIY deployments [code-server install and docs](https://coder.com/docs/code-server/install) [openvscode-server repo](https://github.com/gitpod-io/openvscode-server).
**Integrated terminal / interactive shell** — Both expose a full integrated terminal backed by a shell on the VPS using WebSocket transport; the in-browser terminal provides true interactive shells comparable to Remote-SSH/Remote-Server terminals [code-server docs](https://coder.com/docs/code-server/guide) [openvscode-server repo](https://github.com/gitpod-io/openvscode-server).
**Port binding and exposing admin UIs** — The server listens on a configurable port (code-server defaults to 8080; openvscode-server typically 3000) and can sit behind a reverse proxy (NGINX, Caddy) for TLS, domain routing and authentication. You can also publish additional services on other ports and map them with the reverse proxy or a tunnel [code-server guide](https://coder.com/docs/code-server/guide) [openvscode-server deployment notes](https://github.com/gitpod-io/openvscode-server).
**Extensions, workspace parity and resource overhead** — Both provide much of VS Code’s functionality (extensions, debugging, settings). code-server often provides additional management features and packaging for production deployments and may include marketplace proxying or Open VSX integration; openvscode-server stays closer to upstream VS Code and can be simpler to keep in sync with upstream releases. Both require moderate server resources when many users or heavy extensions run concurrently [code-server FAQ](https://coder.com/docs/code-server/FAQ) [openvscode-server repo](https://github.com/gitpod-io/openvscode-server).
**Proxy and corporate HTTPS inspection behavior** — Both rely on WebSockets; corporate HTTPS-inspecting proxies that break WebSocket upgrades or rewrite headers will disrupt the terminal and live features. The recommended pattern is to terminate TLS at a trusted reverse proxy with proper WebSocket proxying support or to front the service with Cloudflare Tunnel/Access so clients only speak standard HTTPS to Cloudflare’s edge [code-server WebSocket notes](https://github.com/coder/code-server/issues/1817) [openvscode-server proxy discussion](https://github.com/gitpod-io/openvscode-server/issues/163).
**Suitability as a fallback for Remote Tunnels being blocked** — Running a browser-served IDE behind Cloudflare Tunnel + Access or on a public TLS hostname with Access is a robust fallback when vendor relay domains for VS Code Remote Tunnels are blocked; you give up the vendor relay’s integrated auth flow but retain a native-like editor experience in-browser [code-server docs](https://coder.com/docs/code-server/guide).
### Browser-based SSH terminals: ttyd, wetty, sshwifty
**How they work** — These projects run a small web server on the VPS that accepts HTTPS/WebSocket connections and launches an SSH client or shell on the server side; the browser UI uses xterm.js (or equivalent) to emulate a terminal. They are focused on terminal access rather than full IDE features [ttyd repo](https://github.com/tsl0922/ttyd) [wetty repo](https://github.com/butlerx/wetty) [sshwifty repo](https://github.com/nirui/sshwifty).
**Interactive shell support** — All three reliably provide interactive shell sessions. ttyd is C-based and extremely lightweight, wetty runs on Node and supports SSH auth modes, and sshwifty provides additional features like Socks5 backend support and multiple auth modes [ttyd repo](https://github.com/tsl0922/ttyd) [wetty repo](https://github.com/butlerx/wetty) [sshwifty repo](https://github.com/nirui/sshwifty).
**Admin UI on a custom port, editor workspaces and HTTP previews** — These tools do not provide code editor workspaces or HTTP preview forwarding as first-class features. They can listen on configurable ports for the web UI itself; to expose an admin UI you would run the admin UI process separately and either reverse-proxy both services or tunnel the admin UI via Cloudflare/Tailscale/Funnel [ttyd repo](https://github.com/tsl0922/ttyd) [wetty repo](https://github.com/butlerx/wetty).
**Proxy behavior and corporate HTTPS inspection** — Because these terminals use WebSockets over HTTPS, strict proxies that block WebSocket upgrades or intercept the TLS session will break connections. Running them behind Cloudflare Tunnel/Access or a properly configured TLS-terminating reverse proxy that supports WebSocket upgrades is the practical workaround in locked-down clients [community notes on proxy & websockets](https://community.cloudflare.com/t/using-cloudflared-behind-https-proxy/637037).
**Resource trade-offs** — ttyd is low-overhead and ideal for many concurrent lightweight terminals on modest VPSs; wetty and sshwifty are heavier (Node or Docker-based) but provide richer SSH client behavior (public key authentication, config-based connections, Socks5 backends) [ttyd repo](https://github.com/tsl0922/ttyd) [wetty repo](https://github.com/butlerx/wetty) [sshwifty repo](https://github.com/nirui/sshwifty).
### Server-side Tailscale: Funnel and Serve (no client install required for browser-only exposures)
**How it works** — Install the Tailscale client on the VPS and use Serve to publish services within the tailnet or Funnel to expose a local service as a public HTTPS endpoint. Funnel generates a stable HTTPS URL (backed by certificates provisioned automatically) that routes traffic through Tailscale’s relay infrastructure to the VPS; users do not need a Tailscale client to access a Funnel URL in a browser [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel) [Tailscale Serve docs](https://tailscale.com/docs/features/tailscale-serve).
**Interactive shell and SSH** — Tailscale provides Tailscale SSH for authenticated SSH access within the tailnet; that requires Tailscale credentials on the client side to join the tailnet. For browser-only users, Funnel alone does not directly provide SSH in-browser, but you can expose a browser-based SSH terminal (e.g., wetty) on the VPS via Funnel so the user reaches it via an HTTPS Funnel URL without installing a client [Tailscale Serve and Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
**Custom ports, editor hosting, and HTTP previews** — Funnel supports mapping local TCP/HTTP ports to public HTTPS URLs; it is suitable for exposing code-server, openvscode-server, admin UIs and preview URLs. Funnel limits what ports it will serve (commonly ports 443, 8443 and 10000 for direct Funnel endpoints) and Serve/Funnel cannot simultaneously expose the same port in both modes without careful configuration; check the current Funnel port constraints before planning [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
**Corporate proxy behavior** — Funnel traffic routes through Tailscale relay servers and remains end-to-end encrypted; where corporate proxies allow outbound HTTPS to public endpoints, Funnel URLs typically work. If corporate network policy blocks unknown outbound relays or inspects and blocks connections to Tailscale relay endpoints, it may fail. In many locked-down laptop scenarios the browser connecting to a public Funnel HTTPS URL will succeed because it looks like standard HTTPS to a public hostname [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
**Operational caveats** — Funnel makes it easy to expose services without client installs, but Funnel URLs are public by default (unless you add your own gating layer on the app) and you must manage access carefully. Tailscale also exposes identity headers when Serve is used inside a tailnet, which can let the backend validate authenticated requests when clients are on Tailnet [Tailscale Serve docs](https://tailscale.com/docs/features/tailscale-serve).
### Decision matrix: per-access-need recommendations and fallbacks
**Interactive shell access** — Browser-only recommended: Cloudflare Tunnel + Access to a browser SSH terminal (Cloudflare’s browser SSH or an origin web-SSH like ttyd/wetty behind the Tunnel). Cloudflare Access provides a browser-friendly OTP/SSO gate and Cloudflare’s edge maximizes survival through corporate HTTPS proxies [Cloudflare browser SSH docs](https://developers.cloudflare.com/cloudflare-one/networks/connectors/cloudflare-tunnel/use-cases/ssh) [Cloudflare Access OTP](https://developers.cloudflare.com/cloudflare-one/integrations/identity-providers/one-time-pin).
Client-install recommended: Native SSH (OpenSSH / PuTTY) when allowed, or Tailscale client + Tailscale SSH for identity-managed SSH (better auth and audit) [Tailscale SSH docs](https://tailscale.com/docs/features/tailscale-ssh).
Fallback order when browser-only is required: 1) Cloudflare Tunnel + Cloudflare Access → browser SSH; 2) Tailscale Funnel exposing wetty/ttyd on HTTPS; 3) vendor Remote Tunnels ([vscode.dev](http://vscode.dev)) only if relay domains are reachable; 4) browser web-SSH behind direct VPS TLS if Cloudflare/Tailscale are blocked.
**Web-based admin UI on custom/non-standard ports** — Browser-only recommended: Cloudflare Tunnel + Cloudflare Access with a hostname mapped to the origin service (map nonstandard local port to a standard hostname and let Cloudflare proxy it), because clients only need HTTPS to Cloudflare’s edge and Access can protect the UI [Cloudflare Tunnel setup](https://developers.cloudflare.com/tunnel/setup).
Client-install recommended: SSH port-forwarding from a native client or Tailscale Serve inside a tailnet if you prefer identity-aware access and internal-only exposure [Tailscale Serve docs](https://tailscale.com/docs/features/tailscale-serve).
Fallback order: 1) Cloudflare Tunnel + Access; 2) Tailscale Funnel (if Funnel supports required ports and you accept Funnel semantics); 3) vendor Remote Tunnels port-forwarding if relay domains are allowed; 4) expose via HTTPS with a reverse proxy and short-lived certificates on a public host.
**Code editor workspace (VS Code-like experience)** — Browser-only recommended: host code-server or openvscode-server on the VPS and front it with Cloudflare Tunnel + Access. This combination gives a near-native VS Code experience in the browser and survives strict client policies because clients speak HTTPS to Cloudflare [code-server docs](https://coder.com/docs/code-server/guide) and [Cloudflare Tunnel docs](https://developers.cloudflare.com/tunnel/setup).
Client-install recommended: VS Code Remote SSH/Remote Tunnels if client software is allowed (Remote SSH uses native SSH; Remote Tunnels gives integrated vendor relay experience but is fragile in restrictive corporates) [VS Code Remote Tunnels docs](https://code.visualstudio.com/docs/remote/tunnels) [VS Code Remote SSH docs](https://code.visualstudio.com/docs/remote/ssh).
Fallback order: 1) code-server/openvscode-server behind Cloudflare Tunnel + Access; 2) vendor Remote Tunnels ([vscode.dev](http://vscode.dev)) if relay domains reachable; 3) Tailscale Funnel exposing code-server HTTPS; 4) native Remote SSH for installed-client users.
**HTTP preview URLs (development previews / forwarded ports)** — Browser-only recommended: Cloudflare Tunnel for public HTTPS preview URLs protected by Cloudflare Access; Cloudflare Pages preview with Tunnel is a first-class workflow for short-lived previews and survives HTTPS-inspecting proxies because clients talk to Cloudflare’s edge [Cloudflare Pages preview docs](https://developers.cloudflare.com/pages/how-to/preview-with-cloudflare-tunnel) [Cloudflare Tunnel docs](https://developers.cloudflare.com/tunnel/setup).
Client-install recommended: ngrok or localtunnel on the client machine, or Tailscale Serve/Funnel if you prefer server-side control. For locked-down clients, prefer server-side exposures (Cloudflare/Tailscale) rather than local tunneling tools that require client installs [ngrok docs](https://ngrok.com/) [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
Fallback order: 1) Cloudflare Tunnel + Access; 2) Tailscale Funnel; 3) vendor Remote Tunnels forwarded ports (where supported); 4) host previews on a public host with short-lived certs and Access gating.
### Practical VPS provisioning checklist (concrete items to implement before a browser-only workshop)
The following items are server-side tasks operators should do once per workshop image or host. The list is intentionally server-centric so locked-down clients need only a browser.
1) Choose gating and edge strategy — Decide between Cloudflare Tunnel + Access or Tailscale Funnel as the front door. For maximum proxy survivability prefer Cloudflare Tunnel + Access; for direct tailnet features and tailnet identity prefer Tailscale Serve/Funnel. See Cloudflare Tunnel setup guidance [Cloudflare Tunnel setup](https://developers.cloudflare.com/tunnel/setup) and Tailscale Funnel docs [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
2) Install and configure the server agent:
- If Cloudflare: install cloudflared and configure a named tunnel and a route to the service ports you will expose (code-server, admin UI, preview ports) [Cloudflare Tunnel setup](https://developers.cloudflare.com/tunnel/setup).
- If Tailscale: install Tailscale client (latest stable) and enable Serve or Funnel for the ports you will publish [Tailscale Serve docs](https://tailscale.com/docs/features/tailscale-serve) [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
3) Deploy the service(s) on the VPS:
- code-server or openvscode-server (install, systemd service, set port and connection token or password) [code-server install](https://coder.com/docs/code-server/install) [openvscode-server repo](https://github.com/gitpod-io/openvscode-server).
- A browser-based SSH terminal (ttyd/wetty/sshwifty) if you need raw shell access in the browser [ttyd repo](https://github.com/tsl0922/ttyd) [wetty repo](https://github.com/butlerx/wetty) [sshwifty repo](https://github.com/nirui/sshwifty).
4) Configure authentication at the edge:
- Cloudflare: create Access policies and, for workshops, configure one-time PIN/email OTP rules for ephemeral users to reduce provisioning friction [Cloudflare Access OTP](https://developers.cloudflare.com/cloudflare-one/integrations/identity-providers/one-time-pin).
- Tailscale: configure tailnet access policies and Funnel node attributes if you intend to limit who can publish Funnels [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
5) TLS and reverse-proxy considerations (if not using Cloudflare/Funnel):
- Use a reverse proxy (NGINX, Caddy) that supports WebSocket proxying and set trusted certificates from a public CA—or front the site with Cloudflare so the VPS only needs to speak plain HTTP to cloudflared. See guidance for WebSocket/TLS proxying [code-server proxy notes](https://github.com/coder/code-server/issues/1817) and openvscode-server proxy discussions [openvscode-server proxy discussion](https://github.com/gitpod-io/openvscode-server/issues/163).
6) Test from a locked-down client emulation and validate:
- Confirm browsers behind corporate-like HTTPS-inspecting proxies can reach the chosen edge endpoint, authenticate via OTP/SSO, open the integrated terminal (WebSocket upgrades), and load extensions as needed. For Cloudflare, verify that forcing HTTP/2 over TCP/443 works when QUIC/UDP is blocked [cloudflared behind HTTPS proxy community discussion](https://community.cloudflare.com/t/using-cloudflared-behind-https-proxy/637037).
### Security and operational notes (short-lived workshop posture)
**Ephemeral access model** — Use short-lived Access policies or OTP to avoid provisioning long-lived credentials for attendees; Cloudflare Access OTP is specifically built for this use-case [Cloudflare Access OTP](https://developers.cloudflare.com/cloudflare-one/integrations/identity-providers/one-time-pin).
**Least-privilege exposure** — Expose only the specific ports/services required per workshop exercise (editor port, preview ports, admin UI) and gate each with an Access policy or Serve ACL. Avoid funneling generic SSH without identity checks unless you intend open shell access for the exercise [Tailscale Serve docs](https://tailscale.com/docs/features/tailscale-serve).
**Logging and session controls** — Enable edge logging and session recording where available (Cloudflare Access logs, Tailscale audit logs) so you can track who connected during the workshop [Cloudflare Access docs](https://developers.cloudflare.com/cloudflare-one/).
**Be mindful of public exposure** — Tailscale Funnel URLs and Cloudflare hostnames can be public; always combine them with an Access gate (SSO or OTP) and limit session TTLs for workshops [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel) [Cloudflare Access docs](https://developers.cloudflare.com/cloudflare-one/).
### Reliability, scale and common failure modes for time-boxed workshops
**VS Code Remote Tunnels relay fragility** — Although Remote Tunnels provides the most integrated [vscode.dev](http://vscode.dev) experience, workshop-scale usage can surface relay timeouts, stale host sessions and domain-based blocking by corporate firewalls. If you plan many simultaneous attendees and cannot guarantee relay domain accessibility, avoid relying on vendor relay-only flows [VS Code Remote Tunnels docs and issues](https://code.visualstudio.com/docs/remote/tunnels) [relay issue reports](https://github.com/microsoft/vscode/issues/185311).
**Cloudflare Tunnel resilience** — Cloudflare’s edge is resilient to many client-side restrictions; Cloudflare Tunnel + Access is typically the most reliable browser-only route for many concurrent users provided cloudflared has stable outbound connectivity and you respect account limits [Cloudflare Tunnel docs](https://developers.cloudflare.com/tunnel/setup) [account limits](https://developers.cloudflare.com/cloudflare-one/account-limits).
**Tailscale Funnel trade-offs** — Funnel makes server-side publishing simple and often works through corporate proxies because clients see a normal HTTPS endpoint. But Funnel port constraints and public exposure semantics mean you must plan which services to place behind Funnel and which to keep internal [Tailscale Funnel docs](https://tailscale.com/docs/features/tailscale-funnel).
**Browser-served IDEs and terminals** — code-server/openvscode-server and ttyd/wetty/sshwifty are reliable for many simultaneous light-weight users if the VPS has appropriate CPU and memory and the TLS/WebSocket proxying path is stable; place them behind Cloudflare Tunnel or a properly-configured reverse proxy to maximize corporate client survivability [code-server docs](https://coder.com/docs/code-server/guide) [ttyd repo](https://github.com/tsl0922/ttyd).
### Conclusion
Cloudflare Tunnel + Cloudflare Access is the most robust single approach when the client is heavily locked down: it gives browser-only users a familiar HTTPS endpoint, supports OTP/SSO gating for ephemeral workshop access, allows browser-based SSH, and proxies many kinds of web services with good survivability through corporate HTTPS inspection when configured to use TCP/443. When you need a near-native VS Code experience in-browser and the vendor relay domains are allowed, VS Code Remote Tunnels provides the richest integrated experience, but it is more fragile against domain-level blocking and relay outages. Hosting code-server or openvscode-server (or a browser-based SSH terminal) on the VPS and fronting it with Cloudflare Tunnel or Tailscale Funnel gives you the best operational control for workshops: you own the server-side setup, you can tune authentication and quotas, and locked-down clients require only an ordinary browser and successful HTTPS to the chosen public edge.
Implementers should provision the server agents (cloudflared or tailscale), deploy the chosen server-side tools (code-server / openvscode-server / ttyd / wetty), and configure edge Access policies (OTP for attendees or SSO for known users). Validate the full browser flow from representative locked-down clients before the workshop start to catch WebSocket upgrade, QUIC/UDP, or OTP delivery issues and prefer Cloudflare fronting where corporate proxies are known to be strict.
